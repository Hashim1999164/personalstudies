‚ö° Swift Basics

Optionals

Optionals are Swift's way of handling the absence of a value. They're crucial for safe programming.

Value? ‚Üí Optional
Value! ‚Üí Force Unwrap
Value ?? Default ‚Üí Nil Coalescing
var name: String? = nil var age: Int? = 25 // Safe unwrapping if let unwrappedName = name { print(unwrappedName) } // Nil coalescing let displayName = name ?? "Unknown"
Why Optionals Matter: They prevent crashes from nil values. Always use optionals when a value might not exist.

Value Types vs Reference Types

üì¶ Struct (Value Type)

  • ‚úÖ Copied when assigned
  • ‚úÖ Stored on stack
  • ‚úÖ Thread-safe
  • ‚úÖ No inheritance
struct Point { var x: Int var y: Int }

üîó Class (Reference Type)

  • ‚úÖ Shared reference
  • ‚úÖ Stored on heap
  • ‚ö†Ô∏è Need memory management
  • ‚úÖ Supports inheritance
class Person { var name: String init(name: String) { self.name = name } }
Memory Visualization:
Struct: [Value] ‚Üí Copy ‚Üí [New Value]
Class: [Reference] ‚Üí Point ‚Üí [Same Object]

Access Control

private: Only accessible within the same source file. Use for internal implementation details.
fileprivate: Accessible within the same file. Less restrictive than private.
internal: Default access level. Accessible within the same module.
public: Accessible from other modules, but can't be subclassed or overridden.
open: Most permissive. Can be subclassed and overridden from other modules.
// Example class MyClass { private var secret = "hidden" internal var normal = "visible" public var shared = "accessible" open var extensible = "overridable" }
üîó Closures

Closures are self-contained blocks of functionality that can be passed around and used in your code.

Escaping vs Non-Escaping

‚úÖ Non-Escaping (Default)

Must execute before function returns

  • Executes synchronously
  • No @escaping needed
  • Can't outlive function
  • Better performance
func process(completion: () -> Void) { // Executes immediately completion() }

‚è∞ Escaping

May execute after function returns

  • Stored for later
  • Requires @escaping
  • Async operations
  • Need weak self
func fetch(completion: @escaping () -> Void) { // Stored, executes later DispatchQueue.main.async { completion() } }
üí° Remember: Use @escaping when closure is stored or called asynchronously. Always use [weak self] in escaping closures to avoid retain cycles!

Trailing Closures

When a closure is the last parameter, you can write it outside the parentheses.

// Normal syntax someFunction(param: value, closure: { in // code }) // Trailing closure (cleaner!) someFunction(param: value) { // code }

Capture Lists

Capture lists control how values are captured in closures to avoid retain cycles.

class ViewController { var name = "iOS" func setup() { // Weak capture - prevents retain cycle networkCall { [weak self] in guard let self = self else { return } print(self.name) } } }
[weak self]: Creates a weak reference. Self becomes optional. Use when self might be deallocated.
[unowned self]: Assumes self will always exist. Crashes if self is deallocated. Use carefully!
[self]: Strong capture (default). Can cause retain cycles. Avoid in escaping closures.

Array ‚Üí Transform ‚Üí New Array

üó∫Ô∏è Map

Transforms each element

üîç Filter

üìâ Reduce

üîÄ Sort

let sorted = numbers.sorted { $0 > $1 } // [3, 2, 1]
üß† Memory Management
üîÑ ARC (Automatic Reference Counting)

ARC automatically manages memory by tracking references to objects.

How ARC Works:
Object Created ‚Üí Reference Count = 1
New Reference ‚Üí Count++
Reference Removed ‚Üí Count--
Count = 0 ‚Üí Object Deallocated ‚úÖ
class Person { let name: String init(name: String) { self.name = name print("\(name) created") } deinit { print("\(name) deallocated") } } // Reference count = 1 var person1: Person? = Person(name: "John") // Reference count = 2 var person2 = person1 // Reference count = 1 person1 = nil // Reference count = 0 ‚Üí Deallocated! person2 = nil
Key Points:
  • ARC only works with reference types (classes)
  • Value types (structs, enums) are copied, not referenced
  • Strong references are the default
  • Use weak/unowned to break retain cycles
üîó Retain Cycles

A retain cycle occurs when two objects hold strong references to each other, preventing deallocation.

Object A ‚ÜîÔ∏è Object B
(Both holding strong references)
‚ùå Never deallocated!
// ‚ùå RETAIN CYCLE EXAMPLE class Parent { var child: Child? } class Child { var parent: Parent? // Strong reference! } // ‚úÖ SOLUTION: Use weak reference class Child { weak var parent: Parent? // Weak breaks cycle! }
Common Scenarios:
  • Parent-Child relationships
  • Delegates (use weak!)
  • Closures capturing self
  • Notification observers
How to Fix:
  • Use weak for optional references
  • Use unowned when reference always exists
  • Use [weak self] in closures
  • Break cycles manually when needed
‚è≥ Lazy, Weak, Unowned

‚è≥ Lazy Var

Initialized only when first accessed

  • ‚úÖ Delays expensive initialization
  • ‚úÖ Must be var (not let)
  • ‚úÖ Can use self
  • ‚úÖ Thread-safe (Swift 5.0+)
class DataManager { lazy var expensiveData: [String] = { // Only computed when accessed return loadHeavyData() }() }

üîó Weak

Doesn't keep object alive

  • ‚úÖ Prevents retain cycles
  • ‚úÖ Automatically becomes nil
  • ‚úÖ Must be optional
  • ‚úÖ Use for delegates
weak var delegate: ViewControllerDelegate? // Becomes nil if delegate deallocated

üîí Unowned

Assumes object always exists

  • ‚ö†Ô∏è Not optional
  • ‚ö†Ô∏è Crashes if deallocated
  • ‚úÖ Better performance
  • ‚úÖ Use when lifetime guaranteed
unowned let owner: Parent // Assumes owner lives longer
üí° When to Use:
Lazy: Expensive initialization, computed properties
Weak: Delegates, parent references, optional relationships
Unowned: When you're 100% sure object exists (rare!)

MVVM vs MVC

üéØ MVC (Model-View-Controller)

Model ‚Üê‚Üí Controller ‚Üê‚Üí View
  • ‚úÖ Simple and familiar
  • ‚ùå Massive View Controllers
  • ‚ùå Hard to test
  • ‚ùå Tight coupling
// Controller does everything! class ViewController { func fetchData() { } func updateUI() { } func handleTap() { } // 500+ lines of code! }

üöÄ MVVM (Model-View-ViewModel)

Model ‚Üê‚Üí ViewModel ‚Üê‚Üí View
  • ‚úÖ Better separation
  • ‚úÖ Testable ViewModel
  • ‚úÖ Reusable logic
  • ‚úÖ Data binding
// Clean separation! class ViewModel { var data: [Item] = [] func loadData() { } } class View { let viewModel: ViewModel // Just displays data }

Dependency Injection

Passing dependencies from outside instead of creating them internally.

// ‚ùå BAD: Creating dependency inside class LoginViewModel { private let apiService = APIService() // Hard to test! } // ‚úÖ GOOD: Dependency Injection class LoginViewModel { private let apiService: APIServiceProtocol init(apiService: APIServiceProtocol) { self.apiService = apiService } } // Easy to test with mock! let mockService = MockAPIService() let viewModel = LoginViewModel(apiService: mockService)
Benefits:
  • ‚úÖ Easier testing (inject mocks)
  • ‚úÖ Loose coupling
  • ‚úÖ Reusable components
  • ‚úÖ Flexible implementations

Singleton Pattern

Ensures only one instance exists and provides global access.

class NetworkManager { static let shared = NetworkManager() private init() { // Private prevents external creation } func request() { // Network logic } } // Usage NetworkManager.shared.request()
‚ö†Ô∏è Thread Safety: If multiple threads modify singleton, use locks or make it thread-safe.
üí° When to Use: Logging, network managers, shared resources. Avoid for everything!
‚ùå Problems: Hard to test, global state, tight coupling. Use dependency injection instead when possible.
üìê Auto Layout Priorities

Content Hugging Priority

How much a view resists being made LARGER than its intrinsic size.

View wants to stay small
Higher priority = Stays smaller
Lower priority = Can grow more
// Label with high hugging priority label.setContentHuggingPriority(UILayoutPriority.required, for: .horizontal) // Label stays at its text width

Compression Resistance Priority

How much a view resists being made SMALLER than its intrinsic size.

View wants to stay large
Higher priority = Stays larger
Lower priority = Can shrink more
// Label with high compression resistance label.setContentCompressionResistancePriority(UILayoutPriority.required, for: .horizontal) // Label won't truncate text

üìè Content Hugging

Resists being LARGER

  • High = Stays small
  • Low = Can grow
  • Use for: Labels, buttons

üí™ Compression Resistance

Resists being SMALLER

  • High = Stays large
  • Low = Can shrink
  • Use for: Important content

‚öôÔ∏è Concurrency

GCD (Grand Central Dispatch)

üìã Serial Queue

Tasks execute one at a time

let queue = DispatchQueue(label: "serial") queue.async { task1() } queue.async { task2() } // task2 waits for task1

üîÑ Concurrent Queue

Tasks execute simultaneously

let queue = DispatchQueue( label: "concurrent", attributes: .concurrent ) queue.async { task1() } queue.async { task2() } // Both run at same time

Async/Await

Modern Swift concurrency syntax.

// ‚úÖ Modern async/await func fetchData() async throws -> Data { let url = URL(string: "https://api.example.com")! let (data, _) = try await URLSession.shared.data( from: url ) return data } // Usage Task { do { let data = try await fetchData() // Use data } catch { // Handle error } }
Ways to Achieve Concurrency:
  • Threads: Low-level, manual management
  • Dispatch Queues: GCD, high-level abstraction
  • Operation Queues: Built on GCD, more features
üéØ SOLID Principles

SOLID principles guide object-oriented design for maintainable, scalable code.

S - Single Responsibility Principle: One class, one reason to change.
// ‚ùå BAD: Multiple responsibilities class User { func save() { } // Database func sendEmail() { } // Email func validate() { } // Validation } // ‚úÖ GOOD: Single responsibility class User { } class UserRepository { func save() { } } class EmailService { func send() { } }
O - Open/Closed Principle: Open for extension, closed for modification.

Add new features through inheritance/protocols, don't modify existing code.

L - Liskov Substitution Principle: Subclasses should be substitutable for base classes.

Any instance of a subclass should work wherever the base class is expected.

I - Interface Segregation Principle: Many specific interfaces > one general interface.

Clients shouldn't depend on interfaces they don't use. Break large protocols into smaller ones.

D - Dependency Inversion Principle: Depend on abstractions, not concretions.

High-level modules shouldn't depend on low-level modules. Both should depend on abstractions (protocols).

ü§ù Delegation Pattern

Passes responsibility to a delegate object, typically using protocols.

// Protocol defines interface protocol ViewControllerDelegate: AnyObject { func didTapButton() } // ViewController has delegate class ViewController { weak var delegate: ViewControllerDelegate? // Always weak! func buttonTapped() { delegate?.didTapButton() } } // Another class implements delegate class Coordinator: ViewControllerDelegate { func didTapButton() { // Handle action } }
Key Points: Always use weak for delegates to prevent retain cycles. Use protocols to define the interface. Common in UIKit (UITableViewDelegate, etc.).
üîÑ Dependency Inversion

High-level modules shouldn't depend on low-level modules. Both should depend on abstractions.

// ‚ùå BAD: High-level depends on low-level class ViewModel { private let networkManager = NetworkManager() // Concrete! } // ‚úÖ GOOD: Both depend on abstraction protocol NetworkService { func fetch(url: URL) } class ViewModel { private let networkService: NetworkService // Abstraction! } class NetworkManager: NetworkService { } class MockNetworkService: NetworkService { } // Easy to test!

Publisher & Subscriber

üì§ Publisher

üì• Subscriber

Receives and processes values

Manages subscription lifecycle. Store in a property to keep subscription alive.

üåê Networking & Security

SSL Pinning

Validates server certificates against known good certificates.

class SSLPinningDelegate: URLSessionDelegate { func urlSession(_ session: URLSession, didReceive challenge: URLAuthenticationChallenge, completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void) { // Validate certificate against pinned certificate // Only trust if matches } }
Why SSL Pinning: Prevents man-in-the-middle attacks by only trusting specific certificates.

Codable

Type alias for Encodable & Decodable - enables JSON encoding/decoding.

struct User: Codable { let name: String let age: Int } // Encoding let user = User(name: "John", age: 30) let encoder = JSONEncoder() let data = try encoder.encode(user) // Decoding let decoder = JSONDecoder() let decodedUser = try decoder.decode(User.self, from: data)

üì¶ Git & Version Control

Cherry Pick

Applies specific commits from one branch to another.

Branch A: Commit1 ‚Üí Commit2 ‚Üí Commit3
Branch B: (empty)
Cherry-pick Commit2 ‚Üí Branch B now has Commit2 ‚úÖ
# Cherry-pick a commit git cherry-pick <commit-hash> # Cherry-pick multiple commits git cherry-pick commit1 commit2 # Cherry-pick a range git cherry-pick start-commit^..end-commit
Use Cases: Apply bug fixes to multiple branches, selectively merge features, backport changes to older versions.

Stash

Temporarily saves uncommitted changes so you can switch branches or pull updates.

# Save changes git stash # Save with message git stash save "WIP: working on feature" # Apply stashed changes git stash pop # Apply without removing from stash git stash apply # List stashes git stash list # Drop a stash git stash drop stash@{0}
Common Scenario: You're working on a feature, need to switch branches to fix a bug. Stash your changes, switch branch, fix bug, switch back, pop stash. ‚úÖ
üëÅÔ∏è KVO (Key-Value Observing)

Allows objects to observe changes to properties of other objects.

class Person: NSObject { @objc dynamic var name: String = "" } let person = Person() var observer: NSKeyValueObservation? // Observe changes observer = person.observe(\.name, options: [.new, .old]) { object, change in print("Name changed from \(change.oldValue ?? "") to \(change.newValue ?? "")") } person.name = "John" // Observer fires!
Requirements: Class must inherit from NSObject, property must be @objc dynamic.
Modern Approach: Use Combine's @Published or SwiftUI's @StateObject instead when possible.
Cleanup: Always invalidate observers to prevent crashes.
üîÑ Method Swizzling

Exchanges method implementations at runtime. Used for debugging, monitoring, and AOP (Aspect-Oriented Programming).

extension UIViewController { static func swizzleViewDidLoad() { let originalSelector = #selector(UIViewController.viewDidLoad) let swizzledSelector = #selector(UIViewController.swizzled_viewDidLoad) guard let originalMethod = class_getInstanceMethod( UIViewController.self, originalSelector ), let swizzledMethod = class_getInstanceMethod( UIViewController.self, swizzledSelector ) else { return } method_exchangeImplementations(originalMethod, swizzledMethod) } @objc func swizzled_viewDidLoad() { print("ViewDidLoad called on \(type(of: self))") swizzled_viewDidLoad() // Calls original } }
‚ö†Ô∏è Warning: Method swizzling is powerful but dangerous. Use only for debugging/monitoring. Can cause unexpected behavior if not done carefully!
Use Cases: Analytics tracking, logging, debugging, performance monitoring.
Best Practices: Do it once in +load or +initialize, document thoroughly, test extensively.
üéØ OOP Concepts

Polymorphism

Same interface, different implementations. Objects of different types can be treated uniformly.

protocol Shape { func draw() } class Circle: Shape { func draw() { print("Drawing circle") } } class Square: Shape { func draw() { print("Drawing square") } } // Polymorphism in action let shapes: [Shape] = [Circle(), Square()] shapes.forEach { $0.draw() }

Abstraction

Hiding complex implementation details, showing only essential features.

// Abstract interface protocol PaymentProcessor { func processPayment(amount: Double) } // Implementation hidden class StripeProcessor: PaymentProcessor { func processPayment(amount: Double) { // Complex Stripe API calls hidden } }

Inheritance

Classes inherit properties and methods from parent classes.

class Vehicle { var speed: Double = 0 func move() { } } class Car: Vehicle { override func move() { // Car-specific implementation } }
üè≠ Factory Pattern

Delegates object creation to a factory method, hiding creation logic.

protocol Payment { func process(amount: Double) } class CreditCardPayment: Payment { func process(amount: Double) { } } class PayPalPayment: Payment { func process(amount: Double) { } } // Factory class PaymentFactory { static func create(type: String) -> Payment? { switch type { case "credit": return CreditCardPayment() case "paypal": return PayPalPayment() default: return nil } } } // Usage - client doesn't know implementation let payment = PaymentFactory.create(type: "credit")
Benefits: Hides creation logic, reduces coupling, makes it easy to swap implementations, centralizes object creation.
‚öñÔ∏è Static vs Singleton

üìä Static

  • No instance needed
  • Type-level methods
  • Can't be overridden
  • No state storage
  • Thread-safe by default
class Utils { static func calculate() { } } Utils.calculate()

üîó Singleton

  • Single instance
  • Instance-level methods
  • Can maintain state
  • Can implement protocols
  • Need thread-safety
class Manager { static let shared = Manager() private init() {} } Manager.shared.doSomething()

Thread Safety with Singleton

class ThreadSafeSingleton { static let shared = ThreadSafeSingleton() private let queue = DispatchQueue(label: "singleton") private var data: String = "" private init() {} func update(value: String) { queue.sync { self.data = value } } }
üîÑ Control Transfer Statements
return: Exits function immediately, optionally returning a value.
break: Exits loop or switch statement immediately.
continue: Skips remaining code in loop iteration, continues to next iteration.
fallthrough: Continues execution to next case in switch (no automatic fallthrough in Swift).
throw: Throws an error, transfers control to error handler.
// break for i in 1...10 { if i == 5 { break } // Exits loop } // continue for i in 1...10 { if i % 2 == 0 { continue } // Skip even numbers print(i) } // fallthrough switch value { case 1: print("One") fallthrough // Continues to case 2 case 2: print("Two") default: print("Other") }
